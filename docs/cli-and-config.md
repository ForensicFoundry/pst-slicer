## CLI surface

The surface is intentionally tiny and stable. Every entry script is
regenerated by [`bootstrap`](../bootstrap) with a shebang pointing at
the project's uv-managed venv, except for `./ost2pst` which is a
completely stand-alone stdlib-only Python 3 script (no `pst_slicer`
imports; usable from any Python environment):

```
pst-slicer -v | --version                Show version and exit.
pst-slicer -h | --help                   Show help and exit.
pst-slicer <path-to-config.toml>         Run an extraction described by the config.

pst-slicer-baseline -v | --version       Show version and exit.
pst-slicer-baseline -h | --help          Show help and exit.
pst-slicer-baseline                      Chain-of-custody baselining. Compute
    [--verify] [--force]                 SHA-256 + size + mtime for every
    [-o OUT_DIR]                         input file referenced by the configs
    <cfg1.toml> [cfg2.toml ...]          (source PSTs + IMID list files) and
                                         write BASELINE.txt. --verify re-checks
                                         a previously written baseline. See
                                         [Chain-of-custody baselining](./companion-tools.md#chain-of-custody-baselining-pst-slicer-baseline).

pst-slicer-verify -v | --version         Show version and exit.
pst-slicer-verify -h | --help            Show help and exit.
pst-slicer-verify [--no-log]             Run the four-check verification
    <path-to-config.toml>                protocol against the outputs of a
                                         completed extraction that used the
                                         same config. Auto-writes verify.log
                                         to <output.dir>. See
                                         [Post-run verification](./companion-tools.md#post-run-verification-pst-slicer-verify).

pst-slicer-intersect -v | --version      Show version and exit.
pst-slicer-intersect -h | --help         Show help and exit.
pst-slicer-intersect [-o OUT_DIR]        Given two or more IMID-mode configs
    <cfg1.toml> <cfg2.toml> [...]        from parallel PST runs, compute the
                                         IMIDs that were not matched in ANY
                                         run and write the artifacts. See
                                         [Cross-PST intersection](./companion-tools.md#cross-pst-intersection-pst-slicer-intersect).

ost2pst -v | --version                   Show version and exit.
ost2pst -h | --help                      Show help and exit.
ost2pst [--format mbox|eml]              Convert an Outlook OST to a portable
        [--include-deleted] [--force]    mailbox format (MBOX or per-message
        INPUT OUTPUT_DIR                 EML). Stand-alone stdlib-only script.
                                         Wraps readpst with chain-of-custody
                                         hashing and a manifest. See
                                         [OST conversion](./companion-tools.md#ost-conversion-ost2pst).
```

All help / version / error output honors [the palette
rules](#environment-variables). Every tool's `--help` is the
authoritative in-band reference for its exact options.

## Config file schema

The config is TOML. Every path in it may be absolute or relative to
the config file's own directory. Comments (`#`) and blank lines are
allowed anywhere per the TOML spec.

```toml
[input]
pst = "/absolute/path/to/mailbox.pst"      # required

[output]
dir = "/absolute/path/to/output"           # required; created if missing

[mode]
type = "IMID"                              # required; case-insensitive for non-IMID modes

# Exactly one of the following sub-tables, matching mode.type.
```

Supported `mode.type` values (case handling summarized in
[Match modes](./match-modes.md)):

| `mode.type`      | Sub-table required     |
|------------------|------------------------|
| `"IMID"`         | `[mode.imid]`          |
| `"keyword"`      | `[mode.keyword]`       |
| `"sender"`       | `[mode.sender]`        |
| `"sender_domain"`| `[mode.sender_domain]` |
| `"attachment_ext"`| `[mode.attachment_ext]` |

## Environment variables

Color rendering follows [the NO_COLOR spec](https://no-color.org/):

| Variable                    | Effect |
|-----------------------------|--------|
| `NO_COLOR` (any value)      | Force plain output everywhere. Highest priority. |
| `PST_SLICER_NO_COLOR`       | Same as `NO_COLOR` (applies to `pst-slicer`, `pst-slicer-verify`, `pst-slicer-intersect`, `pst-slicer-baseline`). |
| `OST2PST_NO_COLOR`          | Same as `NO_COLOR` (applies to the stand-alone `ost2pst` script). |
| `FORCE_COLOR` (any value)   | Emit color even when stdout/stderr are not TTYs. |
| `PST_SLICER_FORCE_COLOR`    | Same as `FORCE_COLOR` (applies to the `pst-slicer` suite). |
| `OST2PST_FORCE_COLOR`       | Same as `FORCE_COLOR` (applies to `ost2pst`). |

Default (no vars set): color when the target stream is a TTY, plain
otherwise. Files written to disk (manifest, run.log, verify.log,
BASELINE.txt, EMLs) are always plain regardless of the palette. The
auto-written `verify.log` in particular is ANSI-stripped so it is
grep-friendly even when the terminal session was fully colored.

## Exit codes

Exit codes for `./pst-slicer`:

| Code | Meaning |
|------|---------|
| `0`  | At least one message matched and was extracted successfully. |
| `1`  | Extraction error (unreadable PST, IO failure, ...). |
| `2`  | Config / CLI error (bad path, malformed TOML, unknown mode). |
| `3`  | Config was valid and the walk completed, but no message matched. Extraction directory contains an empty manifest, `run.log`, and `not_found.txt`. |
| `130`| Interrupted by user (Ctrl-C). |

Companion tools use their own tool-specific exit code tables; see
[Companion tools](./companion-tools.md) for each tool's individual
codes.
